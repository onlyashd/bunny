01 Building from Source

Prerequisites

  • ✅ Android Studio Otter 3+
  • ✅ JDK 21
  • ✅ Android SDK 29+

Standard build process:

  1. Clone: git clone https://github.com/onlyashd/bunny.git
  2. Open project in Android Studio.
  3. Synchronize Gradle files.
  4. Run the app configuration.

02 System Architecture

Bunny utilizes a Client-Service model to guarantee zero-lag UI performance. By separating the high-intensity RSVP engine from background operations, we ensure that network I/O or heavy parsing never causes a frame drop.

IPC Communication Flow

1. Request
App sends Intent with specific Action via AIDL.
2. Routing
MainService routes via Binder to the logic Implementation.
3. Response
Service returns a Bundle/Primitive across the process boundary.

The Client (App Side)

Uses a reactive ServiceProvider<T> pattern to turn asynchronous AIDL connections into StateFlow<T?>.

// Inject the provider into the ViewModel
@Inject @Configuration
lateinit var configProvider: ServiceProvider<IConfiguration>

// Use it reactively
configProvider.state.collect { service ->
    val bundle = service?.getConfig(Bundle())
    // Update UI with config data
}

The Service (BunnyService Side)

Acts as the Source of Truth. Heavy tasks (Download, Meta-parsing) happen here in a separate process.

  • MainService: The entry point and Intent router.
  • Binder: Maps actions to implementation classes.
  • Impl Classes: Feature-specific logic (e.g., DownloadImpl).

Why this architecture?

RSVP Performance

Downloads and parsing never interrupt the word-flashing engine.

Process Isolation

A crash in the download service doesn't kill the reader app.
